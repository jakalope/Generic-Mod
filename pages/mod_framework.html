Setting up Your Mod Framework

Before I can teach you how to create cool things like custom items, blocks and mobs, we first need to create a skeletal structure for your first upcoming mod. This kind of structure is commonly refer to as a mod framework. The goal of this structure is to deploy a simple modding environment that will be directly connected with Forge API. This environment should be very generic in it's design and should not reflect the personality of your mod, however I will explain this a bit later.

Important
Before you start learning more about how to set up your framework, make sure you've finished my modding introduction guide: Getting Started with Minecraft Modding. Understanding basic concepts being discussed there and getting the actual development environment operational, is an elementary requirement to completing this guide.

Structuring Your Mod

We’ll look at how to organize your mod into different files and what those files should do. This section will be partially based on an official Forge article.

Unlike in my previous guide, this time we will be setting up a mod project from scratch. To do this we will use Eclipse as our IDE. If you prefer another development environment and are asking yourself why I insist on using Eclipse, it's mainly because it's the most popular IDE out there, and most developers already have experience working with it.

Java Project

Your first step should be to create a new workspace folder in an easily accessible location, like Libraries\Documents. Name the folder any way you want. This is where Eclipse will store it's metadata files related to Java projects created and managed inside that workspace. These files will be hosted in two folders called .metadata and .recommenders, so don't mess around with those folders.

Now open Eclipse and select the new folder you just created as your workspace directory. You should have been prompted to selected your workspace by Eclipse launcher. In case this didn't happen, do this - File → Switch Workspace → Other....

Next, we're going to create a new Java project - File → New → Java Project. This will open a new window with several different options. First and foremost, give the project a cool name, something like: "best-minecraft-mod-ever".

Note
When choosing a name for your project, use hyphens or underscores instead of spaces. This is a commonly accepted repository naming convention in the GitHub community, and that's important because later on we will be turning this project into a Git repository.

Select the latest JavaSE as your execution environment JRE (version 1.8 at the moment of writing this article). If you're interested in developing for older versions of Minecraft, you have to find out which execution environment that specific version uses, and select that.

Modify the project layout, by selecting the Create separate folders for sources and class files option. This option will instruct Eclipse to build the project from a dedicated folder containing your code and other resources. We need to create two source folders for this project: src/main/java and src/main/resources. The first folder will hold the main code, while the second one will hold model files, textures and other similar files. To specify a default source folder to use when creating new projects, click the Configure default link and enter the name of the first source folder. Click next and create the second source folder. Click finish and your basic structure is now complete.

Packages

We're going to be organizing our classes by placing them in packages. In case you don't understand what Java packages are, I strongly recommend reading this short Java tutorial.

“
A package is a grouping of related types providing access protection and name space management. Note that types refers to classes, interfaces, enumerations, and annotation types.

Pick a unique package name. The conventional naming scheme starts with your name, followed by the name of your mod like this mandy.amazingmod. However, if you own a URL associated with your project, you can use it as your top level package. For example if you own the “example.com”, you may use com.example as your top level package.

Note
The commonly accepted package naming convention is to define all names in lower case to avoid conflict with the names of classes or interfaces.

Once you've decided on a good name, create the top level package with that name. Now we're going to create two package within the top level package you just created by using a common sub-package strategy recommended by Forge. All code run on server/client should be placed in common package, and client side only code in client package.

By keeping your code in clean sub-packages, you can grow your mod much more organically.

We will begin by defining a framework entry point with the @Mod annotation. Any class found with this annotation applied will be loaded as a Mod. The instance that is loaded will represent the mod to other Mods in the system.
